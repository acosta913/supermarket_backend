# Supermarket Backend - Architecture Continuity Guide

This guide documents the architecture baseline implemented in this project and how to replicate it as the codebase grows.

## 1) Current architectural baseline

- Vertical modules by domain (`products`, `users`, `auth`, `inventory`, `sales`).
- Cross-cutting components under `src/common`.
- Global request tracing with `x-request-id`.
- Standardized error and response handling with global filter/interceptors.
- `products` module uses dependency inversion via repository port + token + Prisma implementation.

## 2) Folder conventions

Use this convention in every module:

```text
src/modules/<module-name>/
  domain/
    entities/
    ports/
  infrastructure/
    repositories/
  dto/
  <module-name>.module.ts
  <module-name>.service.ts
  <module-name>.controller.ts
  <module-name>.tokens.ts
```

Global shared code:

```text
src/common/
  constants/
  filters/
  interceptors/
  middleware/
  types/
  utils/
```

## 3) Cross-cutting pattern to replicate

### Middleware (`RequestIdMiddleware`)
- Read incoming `x-request-id` or generate one.
- Attach it to request and response headers.
- Use it for logs and error traceability.

### Exception filter (`HttpExceptionFilter`)
- Convert all exceptions to a consistent error payload:
  - `error.code`
  - `error.message`
  - `error.details`
  - `error.traceId`

### Interceptors
- `LoggingInterceptor` logs method, URL, status code, elapsed milliseconds, and request id.
- `ResponseInterceptor` wraps plain handler output into a standard `{ data, traceId }` response.

### Utilities
- `pagination.util.ts` centralizes page/limit normalization and metadata generation.
- `prisma-error.util.ts` maps common Prisma errors (`P2002`, `P2025`) to HTTP exceptions.

## 4) Domain + repository pattern (SOLID)

Each module should define:

1. Domain entities (pure TypeScript types).
2. Repository port interface in `domain/ports`.
3. Repository implementation in `infrastructure/repositories`.
4. Token constant in `<module>.tokens.ts`.
5. Service depends on the port (not Prisma directly).

This applies Dependency Inversion:
- high-level logic (service) depends on abstractions.
- low-level database code (Prisma repository) depends on same abstractions.

## 5) How to create a new scalable module

For module `<x>`:

1. Create `domain/entities/<x>.entity.ts`.
2. Create `domain/ports/<x>-repository.port.ts`.
3. Create `<x>.tokens.ts` with `X_REPOSITORY`.
4. Implement `infrastructure/repositories/prisma-<x>.repository.ts`.
5. Register provider in `<x>.module.ts`:
   - `{ provide: X_REPOSITORY, useClass: PrismaXRepository }`
6. Inject token in service:
   - `constructor(@Inject(X_REPOSITORY) private readonly repository: XRepositoryPort) {}`
7. Keep controllers thin: validate/parse input and call service.
8. Keep business rules in service.
9. Keep DB querying/mapping in repository.

## 6) DTO and response standards

- Input DTOs:
  - Use `class-validator` + `class-transformer`.
  - Add Swagger decorators (`@ApiProperty`, `@ApiPropertyOptional`).
  - Validate pagination/sorting with explicit enums.
- Response DTOs:
  - Do not expose raw Prisma objects.
  - Map domain entity -> response DTO in service/mapper.

## 7) Pagination standard

Implement list endpoints with:
- query DTO (`page`, `limit`, `search`, `sortBy`, `order`)
- repository returns `{ items, total, page, limit }`
- service returns `{ data, meta }` where `meta` is generated by `buildPaginationMeta`.

## 8) Error handling standard

- Repository layer catches Prisma errors and maps via `throwMappedPrismaError`.
- Service throws business exceptions (`NotFoundException`, `ConflictException`) when needed.
- Global filter enforces final error payload format.

## 9) Code quality and strong typing roadmap

Gradually harden `tsconfig.json`:

1. `noImplicitAny: true`
2. `strictBindCallApply: true`
3. `noFallthroughCasesInSwitch: true`
4. `noUncheckedIndexedAccess: true`
5. `exactOptionalPropertyTypes: true`
6. `strict: true`

Recommended ESLint hardening:
- `@typescript-eslint/no-floating-promises`: `error`
- `@typescript-eslint/no-misused-promises`: `error`
- `@typescript-eslint/consistent-type-imports`: `error`

## 10) Testing strategy for continuity

Per module:
- Unit tests for service (mock repository port).
- Integration tests for Prisma repository.
- E2E tests for controller contracts.

CI sequence:
1. `npm run lint`
2. `npx tsc --noEmit`
3. `npm test`
4. `npm run test:e2e`

## 11) PR rollout strategy

Use small, auditable PRs:

1. Cross-cutting (`common/` + app bootstrap wiring).
2. Refactor one module to port/repository (`products` already done).
3. Replicate to `inventory`.
4. Replicate to `sales`.
5. Replicate to `users`.
6. Introduce auth guards and role policies.
7. Tighten TypeScript/ESLint and fix incrementally.

## 12) Common anti-patterns to avoid

- Injecting Prisma directly into every service.
- Duplicating pagination and error mapping logic in multiple modules.
- Returning raw ORM objects from controllers.
- Controllers containing business logic.
- Growing "god services" with mixed concerns (HTTP + business + data).
